## 三、技术架构决策

### 最终选型（2026-06-19 确定）

```
Tauri v2 (Rust + WebView2)
  ├── WebView ← frontend/ (HTML/CSS/JS + Chart.js)
  ├── Tauri Plugins ← 系统托盘/自启/全局热键
  └── Sidecar ← Python backend/main.py (数据采集层)
       ├── psutil → 本机系统 (CPU/内存/磁盘/网络)
       ├── WMI → GPU/温度/远程设备
       └── aiohttp → Deepseek API
```

### 决策理由（已确认）

1. **Tauri > Electron**: Surface Go 4GB RAM，Tauri 省 100MB+。Electron 的 Chromium 单独占用 150-200MB，WebView2 是系统组件仅 20-40MB
2. **Tauri > PyQt6**: 前端 HTML/CSS/JS 代码零改动，Chart.js 的实时可视化质量远高于 QChart，开发速度快 5x
3. **Python 后端不变**: psutil / WMI / Deepseek SDK 只有 Python 版，Sidecar 模式是最小侵入方案
4. **纯 Web > 桌面套壳**: 浏览器打开 `localhost:8080` 仍然可用，Tauri 只是增强体验（托盘/自启/全屏）

### 数据流

```
Python (每秒) → WebSocket → Chart.js 实时更新
Python (每30秒) → SQLite 持久化 → Deepseek 历史查询
Frontend (用户操作) → HTTP API → Python (配置保存/CSV导入)
```

## 四、核心边界与约束

### 硬件约束
- Surface Go 第一代: Intel Pentium Gold, 4GB RAM, 10" 1280×800
- 等效视距: 物理 40cm ≈ 27" 显示器 2m → **大字号高对比**
- 触屏操作: 所有交互热区 ≥ 44px

### 设计约束（来自错题本）
1. 不简陋、不浮夸、不赛博 — 构成主义的精心构图 vs 终端面板/霓虹灯
2. 红/黑/白三色系统占 90%+，绿色/黄色仅用于状态指示（<5%）
3. 红色永远不作大面积背景
4. 直角优先 (`border-radius: 0`)，硬阴影（无模糊）
5. 图表最多三色（红/黑/灰）
6. 标题全部大写 ALL CAPS，等宽字体用于数值

### 技术约束
- Python 3.10+，不使用 GPU/NVIDIA 专有库
- 前端无框架依赖（纯原生 JS + Chart.js CDN）
- WebSocket 端口 8765，HTTP 端口 8080（可配置）
- Deepseek API: 官方 `https://api.deepseek.com`，Key 存在 `config.json`
- ⚠️ Deepseek 无公开用量监控 API — `/dashboard/usage` 是非公开接口，需改为本地拦截模式
- 主力机数据通过 WMI 远程获取（后续配置）

## 五、关键流程片段

### 启动流程
```python
# backend/main.py — 当前入口
startup:
  1. init_db()          # SQLite 建表
  2. create_task(system_loop)     # 每秒系统数据 → WebSocket
  3. create_task(deepseek_loop)   # 每30秒 Deepseek → WebSocket + SQLite

# 改 Tauri 后流程:
Tauri 启动 →
  1. 拉起 Python sidecar (backend/main.py)
  2. 等待 WebSocket 连接就绪
  3. 加载 WebView → frontend/index.html
  4. WebView 内 JS 连接 ws://localhost:8765
```

### 数据采集流程
```
collect_system_loop()
  ├── psutil: CPU/内存/磁盘/网络/电池
  ├── WMI: GPU/温度
  ├── 计算网速差值
  └── broadcast({"type": "system", "data": ...})

collect_deepseek_loop()
  ├── GET /user/balance → 余额 (公开API ✅)
  ├── GET /dashboard/usage → Token 用量 (⚠️ 非公开内部接口)
  ├── SQLite: 持久化余额 + 用量
  ├── SQLite: 汇总今日/本周/本月/分模型
  └── broadcast({"type": "deepseek", "data": {...}})

⚠️ 架构风险 (2026-06-20):
  Deepseek 公开 API 仅 4 个端点 (chat/completions, beta/completions, /models, /user/balance)。
  不存在公开的 Token 用量聚合查询 API。当前使用的 /dashboard/usage 是 Deepseek Web
  控制台内部接口，无 SLA 保障。

  正确路线: 在 chat/completions 调用链路中截获 usage 字段 →
  POST 到 Pulse 本地 /api/deepseek/log 端点 → SQLite 持久化。
```

### CSV 导入流程
```
用户拖拽/选择 CSV → 前端解析列名映射 →
POST /api/csv/import (multipart) →
Pandas 读 CSV → 字段匹配（7个标准列） →
批量 INSERT → 返回导入数量 → 前端刷新历史图表
```

### WebSocket 消息格式
```json
// 系统数据（每秒）
{"type": "system", "data": {"cpu": {...}, "memory": {...}, "disk": [...], "gpu": [...], "network_speed": {...}, ...}}

// Deepseek 数据（每30秒）
{"type": "deepseek", "data": {"balance": {...}, "today": {...}, "week": {...}, "month": {...}, "model_breakdown": [...], "history_7d": [...], "limits": {...}, "over_limit_daily": false}}
```

### 前端页面结构
```
Tab 仪表盘:
  ├── 告警条 (超出限额时显示)
  ├── Hero Grid: 4 个 KPI 卡片 (Token/消费/余额/模型)
  ├── Charts Grid: 趋势图 + 分模型柱状图 + 缓存率环形图 + 消费折线图
  └── 系统状态条 (底部, CPU/内存/磁盘/GPU/温度/网络)

Tab 系统:
  ├── CPU+内存实时折线图
  ├── GPU 环形图
  ├── 磁盘柱状图
  └── 系统信息

Tab 历史:
  ├── 30日 Token 趋势图 (双轴: Token + 消费)
  └── CSV 拖拽上传 + 列映射预览 + 确认导入

Tab 设置:
  ├── Deepseek API Key 配置
  ├── 消费限额 (日/月)
  ├── WMI 远程设备
  └── 关于信息
```
