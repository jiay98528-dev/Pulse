## 1. 执行摘要

### 1.1 一句话定位

**GaugePane 仪窗是面向 Windows 桌搭玩家与 AI 开发者的本地运行仪表窗。**

它不是企业监控 SaaS，不是安全监控软件，也不是单纯的 AI 账单工具。它的核心价值是：在低功耗 Windows 副屏上，以低资源占用、可主题化、可长时间常驻的方式展示本机系统负载、AI 成本/余额、Codex 状态、局域网设备和插件状态。

### 1.2 当前阶段

项目已具备较完整的产品雏形：

| 模块 | 当前状态 | 商业意义 |
|---|---|---|
| 本地系统遥测 | 已实现 CPU、内存、磁盘、网络、电池、GPU/WMI 部分数据 | 桌搭玩家和低配副屏用户的基础价值 |
| Dashboard Canvas | 已接入 6 个实时 telemetry canvas | 形成区别于传统表格监控工具的视觉记忆点 |
| AI Provider | DeepSeek 余额/用量为 active，OpenAI/Anthropic 当前为 saved-only | AI 开发者差异化入口，但不能夸大为全供应商遥测 |
| Codex 联动 | 只做本机 CLI/登录状态探针和官方用量入口跳转 | 合规呈现 Codex 状态，不伪造额度查询 |
| 主题系统 | schema v3、主题热更新、导入/导出、市场 UI 已有基础 | 最直接商业化路径 |
| 主题商店 | 独立 8081 服务，已有主题、购买、恢复、支付模拟和邮箱验证能力 | MVP 可验证付费流程，但还不是主安装包内置服务 |
| 发行工程 | 已有 Tauri sidecar、PyInstaller、release smoke 脚本 | 还需完成安装后 smoke、签名、长时运行验证 |

### 1.3 团队与资源

当前团队为中国大陆地区有限公司，两人配置：

| 角色 | 职责 |
|---|---|
| 法人/开发 | 产品定义、核心开发、发行工程、技术内容、社区答疑、质量验收 |
| 商务 | 主题合作、社群运营、支付商户、内容分发、创作者关系、用户反馈整理 |

当前服务器资源：

| 区域 | 配置 | 建议职责 |
|---|---|---|
| 香港 VPS | 1 核 / 1G RAM / 10GB 存储 / 2T 流量 / 1G 带宽 | 主题市场 API、支付回调、邮箱验证码、购买记录主库 |
| 日本 VPS | 1 核 / 1G RAM / 10GB 存储 / 2T 流量 / 1G 带宽 | 静态下载镜像、主题包备份、数据库备份、更新元数据备用节点 |

这套资源足以支撑 MVP 阶段的主题市场、下载页、支付回调和小规模静态分发；不应承诺高可用、高并发、大规模 CDN 或企业级 SLA。

### 1.4 12 个月目标

GaugePane 第一年目标不是直接形成高收入公司，而是验证三个问题：

1. 中国桌搭玩家是否愿意安装一个现代 Windows 副屏仪表盘。
2. 其中一部分用户是否愿意为高质量官方主题支付 ¥5-20。
3. AI/Vibe coding 开发者是否会把 AI 成本与系统负载一起看作一个长期刚需。

第一年关键结果建议设为：

| 时间 | 目标 |
|---|---|
| 30 天 | 完成正式发行闭环、品牌资产、官网落地页、至少 5 个官方主题 |
| 90 天 | 完成首发内容发布，累计 1,000-3,000 下载，验证主题购买闭环 |
| 180 天 | 主题数量 8-12 个，累计 5,000-10,000 下载，形成 1-2 个稳定获客渠道 |
| 365 天 | 累计 10,000-30,000 下载，主题市场产生可复盘收入，决定是否扩展 Pro/插件商业化 |

## 2. 产品定义

### 2.1 核心价值

GaugePane 的核心价值不是“监控更多指标”，而是把用户正在关心的运行状态放到一个可长期常驻、可远距扫视、可个性化表达的仪表窗里。

对于桌搭玩家，它是桌面上的动态仪表装置。  
对于 AI 开发者，它是本机负载和 AI 成本压力的状态面板。  
对于低配副屏用户，它是让旧 Surface Go、旧 Windows 平板继续有价值的常驻工具。

### 2.2 功能边界

| 功能 | v1 口径 | 不应夸大的部分 |
|---|---|---|
| 系统负载 | 本机 CPU、内存、磁盘、网络、部分 GPU/电池实时数据 | 当前不是 AIDA64 级全传感器覆盖 |
| AI 成本 | DeepSeek active；CSV/ZIP 导入与历史分析；OpenAI/Anthropic 先保存配置 | 不声称所有供应商余额/用量都已接入 |
| Codex 状态 | 检测本机 Codex CLI、版本、登录状态，跳转官方用量入口 | 不读取 auth.json；订阅额度只引导用户前往官方入口，不使用 API Key 做查询 |
| LAN 设备 | 插件架构、配对、设备横栏、远程采集方向已有基础 | WMI 远程实机配对仍需后续验证 |
| 主题系统 | 本地主题、热更新、导入导出、市场浏览、购买恢复流程 | 8081 主题商店是外部服务，不打入主安装包 |
| 发行 | Tauri + Python sidecar 构建链基本完成 | 还需安装后 smoke、代码签名、长时 soak test |

### 2.3 对外信息架构

对外官网和首发内容建议采用三层表达：

1. **第一层：仪表窗**  
   “把旧 Windows 副屏变成漂亮的运行仪表。”

2. **第二层：本地实时数据**  
   “CPU、内存、网络、AI 余额、Codex 状态和局域网设备，一眼扫视。”

3. **第三层：主题市场**  
   “像换桌面组件一样更换仪表主题，官方精品主题一次性购买。”

## 3. 目标用户

### 3.1 首发用户：桌搭玩家

这是首发优先人群。原因是他们对视觉、屏幕利用率、设备复活、桌面氛围有强烈兴趣，愿意为了“好看且能常驻”的工具尝试新软件。

典型用户画像：

| 用户 | 场景 | 需求 |
|---|---|---|
| 旧平板复活用户 | 手里有 Surface Go、旧 Windows 平板、小副屏 | 想让闲置设备变成桌面仪表盘 |
| 桌面美化玩家 | 关注小红书/B站桌搭、RGB、状态屏 | 想要更现代、更容易配置的动态面板 |
| 轻度硬件玩家 | 不想折腾 Rainmeter/AIDA64 SensorPanel | 想开箱即用地看到负载和温度趋势 |

首发内容应该突出“漂亮、轻量、能跑在旧设备上”，而不是先讲 AI telemetry。

### 3.2 第二用户：AI / Vibe coding 开发者

这是 GaugePane 的差异化人群。桌搭工具很多，但把 AI 成本、DeepSeek 用量、Codex 状态和系统负载放在同一个本地面板里的工具很少。

典型用户画像：

| 用户 | 场景 | 需求 |
|---|---|---|
| Vibe coding 独立开发者 | 同时跑 IDE、浏览器、Codex、DeepSeek/API | 想知道系统是否卡住、AI 成本是否失控 |
| 多模型 API 用户 | DeepSeek、OpenAI、Anthropic、国产模型混用 | 想在本地统一看到余额、历史成本和配置状态 |
| 本地工具型开发者 | 喜欢 Tauri、本地优先、低占用工具 | 关心隐私、离线能力和可控性 |

这一人群更可能贡献反馈、主题、插件和 GitHub stars，但不一定是第一批付费主题买家。

### 3.3 第三用户：低配 Windows 副屏用户

Surface Go 第一代是产品原型设备，也是传播故事的一部分：一个 4GB RAM 的旧设备仍然可以成为稳定、漂亮的运行仪表窗。

这个人群的关键词是：

- 旧设备复用
- 低资源占用
- 不想安装 Electron 大型应用
- 长时间常驻
- 触屏可操作

## 4. 市场与竞品

### 4.1 市场判断

根据当前项目市场调研，GaugePane 位于几个品类的交叉位置：

| 品类 | 用户已经理解什么 | GaugePane 的机会 |
|---|---|---|
| 系统监控 | 用户知道 CPU、内存、网络悬浮窗 | 做得更漂亮、更适合副屏、更可主题化 |
| 桌面美化 | 用户接受 Rainmeter、桌搭屏、状态屏 | 降低配置门槛，提供一键主题市场 |
| AI 用量管理 | 开发者开始关心 token、余额、缓存命中 | 端侧聚合，不做企业 LLM 可观测平台 |
| Tauri 桌面工具 | 轻量桌面应用正在增长 | 用低占用和本地优先形成可信技术标签 |

已验证事实：

- TrafficMonitor、Rainmeter 等工具说明中国用户对桌面状态显示有长期需求。
- AIDA64 SensorPanel 说明可视化仪表盘存在付费意愿，但配置门槛高。
- 低配 Windows 设备上，企业监控方案过重，个人监控工具通常视觉和扩展不足。
- AI 成本监控目前更多在云平台或开发者工具链侧，缺少本地桌面仪表窗形态。

待验证假设：

- 桌搭用户是否愿意从“免费皮肤文化”转向为官方主题小额付费。
- AI 开发者是否会把 GaugePane 放入长期工作流，而不只是尝鲜。
- 主题市场是否能形成稳定创作者供给。

### 4.2 竞品对比

| 竞品 | 强项 | 弱项 | GaugePane 应对 |
|---|---|---|---|
| TrafficMonitor | 中文用户基础大、轻量、免费 | UI 和扩展架构偏旧，信息深度有限 | 不正面攻击，用“副屏仪表窗 + 主题市场 + AI 状态”错位竞争 |
| Rainmeter | 生态庞大、可玩性强 | 配置复杂，安全和质量参差，付费生态受限 | 提供一键安装、官方主题、低门槛编辑器 |
| AIDA64 SensorPanel | 硬件指标深，用户愿意付费 | 价格高、配置门槛高、视觉依赖用户能力 | 先做 60% 传感器覆盖 + 500% 易用性 |
| NZXT CAM | 视觉现代、开箱即用 | 固定视图、可定制弱、资源占用不低 | 用可主题化和低配适配差异化 |
| Zebar | Tauri、Widget、社区潜力强 | 当前偏任务栏/Widget 美化，深度硬件和 AI 聚合不足 | 视为潜在合作/用户来源，必要时做数据 Provider |
| LLM 用量工具 | 对 AI token/成本更专业 | 多面向工程团队或 CLI，不是桌面仪表窗 | 只做个人端侧聚合和可视化，不做企业可观测 |

### 4.3 差异化

GaugePane 的差异化不是单点功能，而是组合：

1. **轻量**：Tauri + WebView2 + Python sidecar，目标设备是 4GB RAM Windows 平板。
2. **漂亮**：Dashboard Canvas 和主题系统形成视觉冲击。
3. **本地**：用户配置、API Key、SQLite 数据优先保存在本机。
4. **可玩**：主题导入、编辑、热更新和市场，适合桌搭传播。
5. **AI 差异化**：DeepSeek 成本、Codex 状态和 AI Provider 配置让产品区别于普通硬件监控。

## 5. 商业模式

### 5.1 v1 收入原则

GaugePane v1 不做订阅，不把主应用直接收费作为第一收入来源。

原因：

- 桌搭和监控工具用户被免费软件长期教育，直接买断会提高获客阻力。
- 当前产品仍需完成正式发行闭环和稳定性验证，不适合一开始承诺付费主应用 SLA。
- 主题市场更符合用户心理：软件免费，审美和个性化付费。

### 5.2 收入来源

| 收入来源 | v1 状态 | 定价 |
|---|---|---|
| 主应用 | 免费 | ¥0 |
| 官方主题 | 首要收入 | ¥6.9 / ¥12.9 / ¥19.9 |
| 社区主题 | 免费 | ¥0，可展示赞助入口 |
| 创作者付费主题 | 后续开放 | 平台抽成 15-20% |
| 定制主题 | 可作为商务试点 | 按项目报价，不进入公开市场定价 |
| Pro 功能/插件 | 12 个月后评估 | v1 不纳入收入承诺 |

### 5.3 主题定价策略

建议三档：

| 档位 | 价格 | 类型 | 目的 |
|---|---:|---|---|
| 入门主题 | ¥6.9 | 单色调/轻量变体 | 降低首次支付门槛 |
| 标准主题 | ¥12.9 | 完整视觉风格 + 多状态细节 | 主力成交价 |
| 精品主题 | ¥19.9 | 强视觉风格 + 专属 Canvas 调色 + 配套壁纸/说明 | 提高客单价，验证高质量主题需求 |

发行前建议至少准备 5 个官方主题：

1. **Gauge Default / 仪窗默认**：产品主视觉，免费。
2. **Dark Bench / 暗室工位**：暗色桌搭，¥6.9 或 ¥12.9。
3. **Signal Glass / 信号玻璃**：更强视觉冲击，¥12.9 或 ¥19.9。
4. **Frost Console / 霜白控制台**：浅色低噪工作台，验证非深色桌搭用户。
5. **Amber Terminal / 琥珀终端**：复古终端仪表，验证怀旧与硬件玩家审美。

之后扩展到 8-12 个官方主题，再开始邀请社区主题。

### 5.4 主题市场不是第一年现金牛

必须明确：在没有大流量之前，主题市场第一年主要是验证付费意愿和形成生态雏形，不应被写成能立即支撑两人工资的业务。

主题收入要有商业意义，至少需要满足其中一个条件：

- 10 万级下载量。
- 高质量主题形成持续复购。
- 主题之外出现插件、Pro 包、定制主题或赞助收入。
- 通过内容平台持续低成本获客。

因此第一年经营目标应是“建立可持续获客与付费闭环”，而不是“立即盈利”。

## 7. 服务器与运营

### 7.1 服务器分工

当前两台服务器配置有限，但足够做 MVP：

| 服务 | 香港 VPS | 日本 VPS |
|---|---|---|
| 官网静态页 | 可部署 | 可镜像 |
| 主题市场 API | 主节点 | 不建议同时写入 |
| SQLite 购买记录 | 主库 | 只做备份 |
| 支付回调 | 主节点 | 备用，不默认暴露 |
| 邮箱验证码 | 主节点 | 备用 |
| 主题包下载 | 小规模可用 | 镜像可用 |
| 安装包下载 | 不建议长期独立承担 | 可备用，但优先 GitHub Release/对象存储 |
| 自动更新元数据 | 可用 | 可镜像 |

### 7.2 推荐部署

香港节点：

- Nginx/Caddy 反代。
- `store.gaugepane.*`：主题市场 FastAPI。
- SQLite 主库：购买记录、主题记录、验证码。
- 每日导出数据库快照。
- 支付回调只开放固定路径，并记录签名验证日志。

日本节点：

- `download.gaugepane.*` 或备用镜像。
- 静态主题包、预览图、安装包摘要。
- 每日拉取香港节点数据库备份。
- 作为香港节点故障时的静态公告页。

### 7.3 资源边界

1C1G / 10GB 存储的机器不能承担所有增长任务：

- 10GB 存储不适合长期保存大量安装包历史版本。
- 1GB RAM 可以跑 FastAPI + SQLite + Nginx，但不适合叠加复杂后台任务。
- 2T 流量看起来充足，但如果安装包较大，实际可支撑下载量会快速下降。
- 不建议在第一年自建复杂 CDN；安装包优先使用 GitHub Releases 或低成本对象存储。

### 7.4 数据与隐私

服务器只应保存：

- 用户购买邮箱。
- 购买记录。
- 主题下载记录。
- 验证码临时记录。
- 必要的支付交易号。

服务器不应保存：

- 用户 AI API Key。
- 本机系统遥测数据。
- Codex 认证文件。
- 用户本机 SQLite 分析数据。
- Windows 系统账号信息。

这应该成为 GaugePane 的信任卖点：**数据尽量在本机，服务器只处理主题市场交易。**

## 8. 团队分工

### 8.1 法人/开发职责

- 产品路线和功能取舍。
- Tauri/Python/前端主开发。
- 安装包、sidecar、release smoke 和签名。
- 主题系统底层和官方主题质量把关。
- 技术内容：B站、知乎、少数派、GitHub release note。
- 用户反馈中的技术问题排查。

### 8.2 商务职责

- 支付商户、邮箱服务、域名、备案/合规资料整理。
- B站/小红书/酷安/社群账号运营。
- 桌搭博主、主题创作者、设计师合作。
- 收集用户问题并分类：安装、使用、主题、支付、建议。
- 维护主题市场内容节奏和创作者沟通。

### 8.3 每周节奏

| 周期 | 固定动作 |
|---|---|
| 每周一 | 确认本周开发目标、内容目标、主题目标 |
| 每周三 | 看下载量、安装失败、反馈 issue、主题购买数据 |
| 每周五 | 发布一个小版本、主题、演示片段或开发日志 |
| 每月末 | 复盘渠道、收入、留存、最常见失败点 |

两人团队必须避免同时推进太多方向。第一年所有工作都围绕三件事：产品稳定、内容获客、主题市场验证。

## 9. 12 个月路线图

### M0：正式发行闭环（0-1 个月）

目标：让用户在无源码、无 Python、无 venv 环境下安装并运行。

必须完成：

- 安装后 smoke：MSI/NSIS 安装路径启动，后端 sidecar 自动启动。
- 关闭应用后后端进程退出。
- `PULSE_DATA_DIR` 写入用户目录，不写安装目录。
- Dashboard canvas、Hardware canvas、URL tab、主题市场离线态通过 smoke。
- 代码签名策略决策：先 unsigned beta，还是购买代码签名证书。
- 对外品牌资产：GaugePane / 仪窗 logo、图标、官网首屏、产品截图。

不做：

- 不做企业账号。
- 不做云同步。
- 不做订阅。
- 不承诺自动更新已经生产可用。

### M1：桌搭首发（1-2 个月）

目标：拿到第一批真实用户。

交付：

- 官网落地页。
- GitHub Release 或下载页。
- 至少 5 个官方主题。
- 2 条 B站视频，5-10 条小红书/酷安短内容。
- 用户反馈表或 issue 模板。

验收：

- 累计下载 1,000+。
- 收到 20+ 有效反馈。
- 至少 5 个用户提供真实桌面截图。

### M2：主题市场 MVP（2-4 个月）

目标：验证主题购买闭环。

交付：

- 香港节点上线主题市场 API。
- 微信/支付宝支付流程从模拟切换到真实可用前的灰度。
- 邮箱恢复已购流程可用。
- 8-12 个官方主题。
- 主题详情页和预览图标准化。

验收：

- 完成真实小额支付闭环。
- 付费主题累计购买 50-150 次。
- 退款/恢复/重复购买逻辑无严重事故。

### M3：创作者冷启动（4-6 个月）

目标：让 GaugePane 不只靠团队自己生产主题。

交付：

- 主题制作指南。
- 主题包校验脚本。
- 社区主题提交模板。
- 创作者展示页。
- 3-5 个外部创作者主题。

验收：

- 社区主题累计 10 个。
- 主题市场月活浏览用户稳定增长。
- 至少 1 个创作者愿意持续更新主题。

### M4：传感器增强与开发者差异化（6-9 个月）

目标：增强产品专业度，避免只被当成桌面装饰。

交付：

- LibreHardwareMonitor 可选接入方案验证。
- GPU 温度、风扇、SMART 硬盘健康等高级指标原型。
- DeepSeek/CSV 分析体验打磨。
- Codex 状态卡继续优化，但不越过合规边界。
- LAN/WMI 实机配对验证。

验收：

- 高级传感器在主力测试设备上稳定。
- 低配设备资源占用仍可接受。
- AI 开发者用户反馈中，“真的有用”占比提高。

### M5：生态与商业扩展（9-12 个月）

目标：决定 GaugePane 是否进入下一阶段。

可选方向：

- 付费主题分成。
- 官方主题包合集。
- 设计师合作主题。
- 轻量插件市场。
- 桌搭品牌/硬件商合作。
- Pro 功能试探，但不默认订阅化。

验收：

- 累计下载 10,000-30,000。
- 至少一个渠道能稳定带来新增用户。
- 主题市场收入覆盖基础服务器和工具成本。
- 团队明确第二年是否继续做主题市场，还是转向 Pro/插件/定制。

## 10. 财务模型

### 10.1 假设

以下为保守估算，不作为承诺：

- 主应用免费。
- 主题单价：¥6.9 / ¥12.9 / ¥19.9。
- 主题平均成交金额按 ¥10-18 区间估算。
- 主题购买转化率按累计下载用户的 1%-3% 粗估。
- 付费用户第一年平均购买 1-2 个主题。
- 不计算订阅收入。
- 不计算企业定制收入。
- 服务器为已拥有资源，仍按运营成本纳入预算。

### 10.2 收入情景

| 情景 | 12 个月累计下载 | 付费转化 | 付费人数 | 人均主题消费 | 主题销售额 |
|---|---:|---:|---:|---:|---:|
| 保守 | 10,000 | 1% | 100 | ¥10-15 | ¥1,000-1,500 |
| 基准 | 20,000 | 2% | 400 | ¥12-18 | ¥4,800-7,200 |
| 较好 | 30,000 | 3% | 900 | ¥15-25 | ¥13,500-22,500 |

解释：

- 这个收入水平不能支撑两人工资。
- 它的意义是验证用户愿意为桌面仪表主题付费。
- 如果 12 个月内无法达到基准情景，说明主题市场作为主收入路径需要调整。
- 如果达到较好情景，可以开始测试主题包、创作者分成或 Pro 插件。

### 10.3 成本预算

| 成本项 | 估算 | 说明 |
|---|---:|---|
| 域名 | ¥60-200/年 | 建议尽快注册 GaugePane 相关域名 |
| 服务器 | 已有资源或 ¥300-1,200/年 | 香港/日本 VPS 作为 MVP 运营 |
| 邮件服务 | ¥0-300/年 | 低量验证码可先用 SMTP |
| 支付通道 | 按交易扣费 | 微信/支付宝商户费率按实际合同 |
| 代码签名 | ¥1,000-4,000+/年 | 是否购买取决于发行策略；会影响 Windows 信任 |
| 设计资产 | ¥0-2,000 | 官方主题、logo、截图、模板 |
| 内容推广 | ¥0-3,000 | 以自制内容为主，少量样机/素材成本 |
| 法务/合规 | 视情况 | 用户协议、隐私政策、商标检索 |

### 10.4 现金流判断

第一年建议按“低成本验证”经营：

- 不租更高规格服务器，除非下载量或商店请求证明需要。
- 不提前采购大量广告。
- 不把主题收入作为人员工资来源。
- 优先投入代码签名、官网、主题质量和内容生产。
- 如果收入无法覆盖基础成本，但下载和反馈增长良好，可以继续以项目资产方式投入。
- 如果下载、留存、付费三项都弱，应在 6 个月时收缩为开源工具或桌搭作品，而不是继续扩大商业投入。

## 11. 风险与对策

### 11.1 发行风险

风险：安装包在无源码、无 Python、无 venv 环境下启动失败。  
对策：M0 必须完成安装后 smoke，并把安装路径启动作为发布前门槛。

风险：未签名应用触发 Windows SmartScreen。  
对策：首发可以标注 beta/unsigned，但正式推广前应评估代码签名证书。

### 11.2 产品风险

风险：用户认为只是“好看的任务管理器”。  
对策：首屏和内容必须展示主题、AI 成本、Codex 状态和副屏常驻场景。

风险：高级硬件指标不如 AIDA64。  
对策：不正面比传感器数量，强调开箱即用、低占用、主题化；后续接入 LHM 增强专业度。

风险：AI 遥测能力被误解。  
对策：文案明确 DeepSeek active，OpenAI/Anthropic saved-only；Codex 只做状态和官方入口。

### 11.3 商业风险

风险：主题付费转化低。  
对策：先用 ¥6.9-19.9 低价测试；如果 6 个月仍弱，转向主题包、定制、插件或赞助。

风险：社区主题冷启动失败。  
对策：前 3-6 个月官方主题自供给，不依赖社区；商务只邀请少量创作者试点。

风险：免费竞品压价。  
对策：免费主应用降低进入门槛，付费只卖审美和便利，不卖基础功能。

### 11.4 合规风险

风险：支付、发票、退款、隐私政策不完善。  
对策：正式收款前准备用户协议、隐私政策、退款规则、支付商户配置和数据留存边界。

风险：商标和域名冲突。  
对策：GaugePane 只是当前候选品牌；发布前做 9/42 类商标精检，尽快注册核心域名。

风险：服务器在香港/日本，面向大陆用户访问波动。  
对策：MVP 可接受；下载增长后引入对象存储/CDN，避免让 1C1G VPS 承担大文件分发。

### 12.5 90 天判断点

| 指标 | 低于该值 | 达到该值 | 超过该值 |
|---|---|---|---|
| 下载量 | <1,000 | 1,000-3,000 | >5,000 |
| 有效反馈 | <10 | 20+ | 50+ |
| 主题购买 | 0 | 20-50 | 100+ |
| 社区截图 | <3 | 5-10 | 20+ |
| 复购/二次打开 | 很低 | 可观察 | 明显 |

如果下载强但付费弱，应继续做免费主应用和主题内容，但重新评估主题价格/质量。  
如果下载、反馈、付费都出现正信号，可以进入 M2/M3 的主题市场和创作者冷启动。

## 13. 结论

GaugePane / 仪窗 的最佳商业切入点不是“企业级 AI telemetry”，而是：

> **把 Windows 副屏变成一个漂亮、轻量、可长期常驻的本地运行仪表窗。**

桌搭玩家给它第一批传播，AI/Vibe coding 开发者给它差异化，主题市场给它第一条可验证收入线。当前团队和服务器资源适合低成本自举，不适合一开始做高可用 SaaS、订阅平台或大规模云服务。

第一年最关键的不是收入规模，而是证明三件事：

1. 用户愿意安装并长期打开。
2. 用户愿意为高质量主题小额付费。
3. GaugePane 能在“桌搭美观”和“开发者实用”之间形成稳定品牌心智。

只有这三件事成立，第二年才值得扩大到插件市场、Pro 功能、创作者分成或更重的商业化路径。
