# Pulse 市场调研报告

> 版本: 1.0 | 日期: 2026-06-21
> 调研范围: 系统监控 × 跨设备监控 × 仪表盘 × AI用量追踪 × 主题市场 × Tauri生态 × 中国市场
> 竞品总数: 50+ 工具/平台，覆盖 7 个类别

---

## 一、摘要

Pulse v2.0 定位为 **"系统硬件监控 + AI 余额聚合 + 可配置组件墙 + 主题市场"四合一桌面看板**，面向 Surface Go (4GB RAM / Pentium Gold) 等低配 Windows 设备。

经过对 50+ 竞品的系统调研，核心结论如下：

| 结论 | 置信度 |
|------|--------|
| **Pulse 的核心定位组合（系统+AI+组件墙+主题市场）在当前市场无直接竞品** | 🔴 高 |
| **低配硬件（4GB RAM）全功能仪表盘细分市场为零竞品状态** | 🔴 高 |
| **Tauri v2 架构在资源效率上对 Electron 有结构性优势（5-8x RAM 差距）** | 🔴 高 |
| **AIDA64 SensorPanel ($60) 验证了可视化仪表盘的付费意愿** | 🟡 中高 |
| **Zebar (Tauri, 30k Stars) 是最接近的潜在竞品，但目前偏 Widget 美化** | 🟡 中 |
| **中国 AI Token 监控市场缺乏本地端侧聚合方案** | 🔴 高 |

#### 3.1.2 关键发现

**AIDA64 SensorPanel — Pulse 最直接竞品对标**
- $60 售价证明用户**愿意为可视化仪表盘付费**
- SensorPanel 可自由布局图表/仪表/文字 → 与 Pulse 组件墙理念一致
- 但配置门槛极高（手写代码级别）→ **Pulse 的拖拽配置是降维打击**
- Pulse 只需做到 SensorPanel 60% 的传感器覆盖 + 500% 的易用性 = 绝对优势

**LibreHardwareMonitor — 技术栈关键发现**
- 提供 NuGet 包 / REST API / WMI 接口
- **Pulse Python Sidecar 可通过 HTTP 直接调用 LHM 获取传感器数据，不需要自己写 Ring0 内核驱动**
- Home Assistant 官方集成、RTSS 插件兼容、FanControl 等项目均依赖 LHM

**NZXT CAM — UI 标杆**
- 此品类中 UI 最精美的工具，**颜值即竞争力**
- 无需账号即可使用 → 验证 Pulse 的邮箱验证策略不会成为获客障碍
- 但 3 种固定视图不可定制 → Pulse 的组件墙差异化明显

**Rainmeter — 社区生态参照**
- 庞大皮肤社区（DeviantArt/GitHub）验证了「主题二创」需求
- 用户最大痛点：安装配置复杂
- Pulse 主题市场需做到：**一键安装 + 零配置 + 即时预览**

**TrafficMonitor — 中国市场验证**
- 44k+ GitHub Stars，中文用户首选
- 但 C++ MFC/Direct2D 架构老旧，不支持跨平台，不支持插件扩展
- **Tauri 方案存在对 TrafficMonitor 实现技术代差攻击的机会**

#### 3.3.3 关键发现

1. **所有 LLM 可观测性工具面向"生产 LLM 应用团队"，而非"个人 AI 用户"** — Langfuse $29-2,499/月，LangSmith $39/座/月
2. **开源社区已涌现一批本地/桌面 Token 追踪工具**，验证了个人 AI 用量监控需求
3. **中国 AI Token 监控集中在云厂商平台侧**，缺少本地端侧聚合多供应商的方案
4. **New API 是最接近 Pulse AI 模块的竞品** — 多供应商 + 成本看板 + 微信/支付宝充值，但它是 API 网关而非桌面看板
5. **Pulse 的差异化：** 追踪"我的多个 AI 账户还剩多少余额"，而非"我的 LLM 应用消耗了多少 Token" — 两种不同场景

#### 3.5.2 Tauri vs Electron — 结构性优势

| 指标 | Electron | Tauri v2 | 优势倍数 |
|------|----------|----------|----------|
| 安装包大小 | 80-200 MB | **2-10 MB** | 10-20x |
| 空闲内存 (RAM) | 150-400 MB | **30-50 MB** | 5-8x |
| 冷启动时间 | 1-5 秒 | **0.3-1 秒** | 3-5x |

**结论：** 在桌面仪表盘（用户全天挂机）场景下，Tauri 的架构优势是结构性的——使用系统原生 WebView 而非捆绑完整 Chromium，Electron 无法逾越。

## 四、市场空白确认

经过对 50+ 竞品的系统排查，以下 **5 项市场空白** 得到了确认：

| # | 空白领域 | 竞品状态 | Pulse 的切入点 | 置信度 |
|---|----------|----------|---------------|--------|
| 1 | **低配硬件 (4GB RAM) 全功能仪表盘** | 零竞品 — 企业方案太重，个人工具功能太浅 | Tauri 30-50MB + HTML/CSS/JS 无框架 | 🔴 确认空白 |
| 2 | **多 AI 供应商本地聚合仪表盘** | 仅云平台自身控制台，无端侧聚合方案 | 分析 Tab + DeepSeek/OpenAI/Anthropic 余额采集 | 🔴 确认空白 |
| 3 | **系统监控 + AI 用量一体化看板** | 竞品要么做系统、要么做 LLM，无融合方案 | 仪表盘 Tab (硬件KPI + AI余额组件) + 硬件Tab + 分析Tab | 🔴 确认空白 |
| 4 | **中国市场现代桌面仪表盘** | TrafficMonitor (功能浅) + Rainmeter (门槛高) | 开箱即用的5 Tab SPA + 中文优先 | 🟡 确认需求 |
| 5 | **带插件架构 + 主题市场的桌面仪表盘** | 仅 Zebar (偏Widget) | PluginBase + 主题热插拔 + 付费市场 | 🟡 确认差异化 |

## 五、Pulse 差异化定位

### 5.1 竞争定位象限

```
                    系统监控深度 ↑
                                   │
                  TrafficMonitor    │   Pulse ★
                  Rainmeter         │  (硬件+AI+插件+主题)
                  Pachtop ⚫        │
                  Stats ⚫          │
                  Glances           │
                                   │
      ── 仅监控 ─────────────────────────── 仪表盘+SaaS ──→
                                   │
                  ActivityWatch    │   RescueTime ($6.5/月)
                  (迁移Tauri中)     │   Toggl Track ($9-18/月)
                                   │
                    个人分析/AI ↓
```

Pulse 是唯一占据 **"系统监控深度 + 仪表盘可定制性 + AI 用量聚合 + 主题市场"** 四象限交叉位置的工具。

### 5.2 核心竞争壁垒

### 5.4 LibreHardwareMonitor 迁移分析

> 决策背景：LHM 提供完整的 REST API/NuGet/WMI 接口，可使 Pulse 传感器从 ~15 暴增到 250+。用户要求评估迁移代价。

**现状 vs LHM 对比：**

| 类别 | 当前 (psutil + WMI) | LHM 迁移后 |
|------|---------------------|-----------|
| CPU | 使用率/每核/频率/负载 | + 每核温度、核心电压、功耗 (Package/IA/GT)、功耗限制原因 |
| GPU | 名称/显存/驱动/刷新率 (WMI) | + **核心/显存温度、风扇转速/百分比、核心/显存频率、功耗、GPU 使用率** |
| 主板 | ❌ 无 | + 芯片组温度、VRM 温度、风扇转速/控制 |
| 存储 | 分区/挂载点/用量 | + **S.M.A.R.T. (健康度/温度/读写量/通电时间)、NVMe 温度** |
| 网络 | 总字节/包收发 | + 每网卡速率（非仅总吞吐） |
| 温度 | psutil 有限 (通常仅 CPU) | + CPU/GPU/主板/硬盘 全面覆盖 |
| 指标总数 | ~15-20 | **250+** |

**迁移代价：**

| 维度 | 详情 |
|------|------|
| **新增依赖** | .NET 6/8 Desktop Runtime (~50MB 磁盘) + LHM DLL (~2MB) |
| **通信方式** | Python HTTP `→` `localhost:8085` LHM Web Server |
| **代码改动** | 新增 `collectors/lhm.py` (~150行)；`system.py` 保留不变；`main.py` 增加一个采集协程 |
| **RAM 增加** | LHM Web Server 空闲 ~15MB（可接受，总内存仍 < 100MB） |
| **分发方式** | LHM MIT 许可证，可嵌入 Sidecar 目录随 Pulse 分发；或通过 winget 自动安装 |
| **启动管理** | Pulse 启动时自动拉起 LHM Web Server；Pulse 退出时自动关闭 |
| **回退策略** | 启动时检测 LHM 可用性 → 可用则启用扩展传感器，不可用则回退纯 psutil |

**好处 vs 坏处：**

| 好处 | 坏处 |
|------|------|
| ✅ 传感器暴增 ~15 → 250+，硬件 Tab 专业度质的提升 | ❌ 增加 .NET Runtime 外部依赖 |
| ✅ GPU 温度/风扇/功耗从无到有（当前完全缺失） | ❌ 额外进程需要生命周期管理 |
| ✅ S.M.A.R.T. 硬盘健康度 — 差异化杀手功能 | ❌ LHM 若停更需自行维护 fork |
| ✅ Home Assistant / FanControl 社区验证 | ❌ 新硬件传感器兼容偶有问题 |
| ✅ MIT 许可证，无限制集成 | |

**推荐方案（两阶段）：**

| 阶段 | 动作 | 风险 |
|------|------|------|
| **Phase 1 (立即)** | 新增 `collectors/lhm.py` 作为可选后端。启动时 HTTP 探测 `localhost:8085`。可用→启用扩展传感器；不可用→回退 psutil。**对不装 LHM 的用户零影响。** | 极低 |
| **Phase 2 (打包时)** | Pulse 安装包内置 LHM portable 版，自动拉起/管理生命周期。用户无需额外安装任何东西。 | 低 |

## 十、Zebar 深度技术分析

> 调研日期：2026-06-21 | 源代码级分析 | 目的：评估竞品威胁 + 探索 API 兼容

### 10.1 架构概述

Zebar 是 pnpm monorepo，包含 3 个核心包：

| 包 | 技术栈 | 职责 |
|---|---|---|
| `packages/desktop` | Rust + Tauri v2 | CLI、原生窗口管理、Provider 后端、系统托盘 |
| `packages/client-api` | TypeScript (npm `zebar`) | Widget 与 Tauri 后端通信的业务逻辑层 |
| `packages/settings-ui` | SolidJS | 内嵌设置界面 / 市场浏览界面 |

**Hub-and-Spoke 架构：** `main.rs` 为中央协调器，每个 Widget 运行在独立原生 WebView2 中（比 Electron 轻量），通过 `zebar` npm 包的 TypeScript API 经 Tauri IPC 桥与 Rust 后端通信。

### 10.2 Provider 系统 — 核心机制

**双运行时：**

| 运行时类型 | 调度方式 | 适用场景 |
|---|---|---|
| Async | `tokio::spawn()` | I/O 密集型（天气、IP查询） |
| Sync | `tokio::spawn_blocking()` | CPU 密集型（系统监控轮询） |

**数据管线（以 CPU Provider 为例）：**
1. CPU Provider 通过 `sysinfo` crate 按配置间隔（默认 5s）采集
2. 通过 `ProviderEmitter` 发射新数据
3. `ProviderManager` 通过 `tokio::broadcast` channel 广播
4. Tauri 事件系统推送到所有订阅该 Provider 的 WebView Widget
5. Widget 的 `onOutput` 回调触发，携带最新数据

**关键限制：Provider 必须编译进 monorepo，不支持 out-of-tree 动态加载。** 第三方贡献需 fork → 写 Rust trait 实现 + TypeScript 镜像 → 提交 PR → 等待合并。

### 10.3 传感器深度对比（关键发现）

Zebar 底层依赖 Rust `sysinfo` crate，这是一个**轻量跨平台系统信息库**，设计目标不是深度硬件监控。

| 维度 | Zebar (sysinfo) | Pulse 当前 (psutil+WMI) | Pulse + LHM (未来) |
|------|----------------|------------------------|-------------------|
| CPU 温度 | ❌ | ⚠️ 有限 | ✅ 每核 |
| CPU 每核使用率 | ❌ (仅整颗) | ✅ | ✅ |
| CPU 功耗 | ❌ | ❌ | ✅ |
| GPU | ❌ **完全没有** | ✅ 名称/显存/驱动 | ✅ **温度/风扇/频率/功耗** |
| 内存 | ✅ RAM + Swap | ✅ | ✅ |
| 磁盘分区 | ✅ 挂载点/容量 | ✅ | ✅ |
| S.M.A.R.T. | ❌ | ❌ | ✅ **健康度/温度/读写量** |
| 风扇转速 | ❌ | ❌ | ✅ |
| 主板传感器 | ❌ | ❌ | ✅ VRM温度/芯片组 |
| 网络速率 | ❌ (仅累计字节) | ✅ 收发字节 | ✅ 每接口 |
| 指标总数 | ~20 | ~15-20 | **250+** |

**结论：Pulse 即使仅用 psutil，在传感器深度上已超越 Zebar（GPU 信息 + 每核 CPU）。引入 LHM 后是数量级差距。**

|---|---|---|
| `cpu` | 全平台 | usage, frequency, logicalCoreCount, vendor |

| `date` | 全平台 | formatted, iso, timezone/locale 可配置 |
| `weather` | 全平台 | isDaytime, status, temperature（依赖外部 API） |
| `ip` | 全平台 | 公网 IP + approxCity/Country/LatLong |

| `audio` | **仅 Win** | 播放/录音设备列表, setVolume(), setMute() |

### 10.6 Pulse × Zebar 集成方案

| 路径 | 描述 | 难度 | 推荐度 |
|------|------|------|--------|
| **A: HTTP 桥接** | Zebar Widget (纯 HTML/JS) 直接 fetch Pulse API (`localhost:8080/api/...`) 或连接 WebSocket。**零侵入，不需修改任何一方源码。** | 极低 | ✅ **强烈推荐** |
| B: shellExec 调用 Python | Widget 通过 `shellExec('python', [...])` 调轻量脚本 | 中 | 🟡 延迟高 |
| C: 贡献 Rust Provider | Fork Zebar monorepo，创建 Pulse Provider（Rust trait + TS 镜像） | 高 | ❌ 维护成本高 |

**推荐方案 A 详解：**
1. Pulse 添加 `/api/zebar/bridge` 端点，返回 Zebar Widget 友好的 JSON 格式
2. 开发一个 Zebar Widget，在 Widget HTML/JS 中 fetch Pulse 数据端点
3. Widget 嵌入 Zebar 工具条/桌面，展示 Pulse 的深度硬件数据
4. **Pulse 成为 Zebar 的"深度数据后端"，Zebar 成为 Pulse 的"任务栏渲染前端"**

### 10.7 威胁评估更新

| 维度 | 分析前 | 分析后 |
|------|--------|--------|
| Zebar 是否是系统监控竞品？ | ⚠️ 有可能是 | **否 — 传感器深度几乎为零** |
| 它能否快速加上深度监控？ | 不确定 | **需 2-4 周专注开发 + 引入 LHM/nvml 等原生库** |

| 风险等级调整 | ⚠️ 中 | **🟢 低 — Zebar 定位是"任务栏美化"，不是"数据看板"** |
